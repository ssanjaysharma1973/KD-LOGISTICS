
# 🏗️ MULTI-CLIENT OPERATION ENGINE - COMPLETE GUIDE

## Architecture Overview

```
┌─────────────────────────────────────────────────────────────┐
│         MULTI-CLIENT E-WAY BILL OPERATION ENGINE             │
├─────────────────────────────────────────────────────────────┤
│                                                               │
│  LAYER 1: Common Master Table                                 │
│  ├─ eway_bill_master_v2 (same for ALL clients)               │
│  ├─ Columns: client_id, eway_bill_no, status, operation...   │
│  └─ Universal statuses for all clients                        │
│                                                               │
│  LAYER 2: Client-Specific Rules Config                        │
│  ├─ client_operation_rules (PER CLIENT)                       │
│  ├─ auto_generate, auto_part_b_update, approval_required     │
│  ├─ gps_required, monitoring_only, multi_vehicle_allowed      │
│  └─ api_mode, transporter_mode, retry_limit, etc.            │
│                                                               │
│  LAYER 3: Logic Engine                                        │
│  ├─ processEwayBill(record)                                  │
│  ├─ Reads client_id → Fetches rules → Determines op type     │
│  ├─ Routes to appropriate handler (GEN, PARTB, TRACK, etc.)  │
│  └─ Executes based ONLY on client config                     │
│                                                               │
└─────────────────────────────────────────────────────────────┘
```

## Operation Types (Automatic Routing)

### Automatic Decision Tree

When a record arrives, the engine automatically determines operation type:

```
New Record Received
    ↓
Load client rules
    ↓
Is client_id known? → No → Use DEFAULT rules
    ↓ Yes
Validate mandatory fields
    ↓
Has eway_bill_no?
    ├─ No  → Needs generation → GEN_PLUS_PARTB (or GEN_ONLY)
    ├─ Yes + vehicle_no empty → PARTB_ONLY
    ├─ Yes + in_transit → TRACK_ONLY
    └─ Yes + near_expiry → EXTENSION_ONLY

Needs approval?
    ├─ Yes → Queue as WAITING_APPROVAL
    └─ No → Execute immediately
```

---

## Operation Types Explained

| Type | When | Example |
|------|------|---------|
| **GEN_ONLY** | Generate EWB only | First document entry |
| **GEN_PLUS_PARTB** | Generate + assign vehicle | Complete shipment prep |
| **PARTB_ONLY** | Update Part-B (vehicle exists) | Vehicle reassignment mid-trip |
| **TRACK_ONLY** | Monitor external EWB | 3PL-generated bills |
| **EXTENSION_ONLY** | Extend expiry | Trip delayed |
| **MULTI_VEHICLE** | Support vehicle changes | Long-haul with relay |
| **MANUAL_APPROVAL** | Wait for admin | High-value shipments |
| **BULK_UPLOAD** | Process batch import | Month-end invoices |

## Universal Status Workflow

All clients use the same statuses:

```
NEW
  ↓
VALIDATED
  ↓
WAITING_APPROVAL (if approval_required=true)
  ↓
READY_FOR_GENERATION
  ↓
GENERATED
  ↓
PARTB_UPDATED
  ↓
IN_TRANSIT
  ├─→ NEAR_EXPIRY (2 hours before expiry)
  │     ├─→ EXTENSION_PENDING
  │     └─→ EXTENDED
  ↓
COMPLETED
  OR
CANCELLED
  OR
ERROR (with error_code)
```

## Database Tables

### Table 1: eway_bill_master_v2
```sql
Common master for all clients - same structure, different rules applied based on client_id

Columns:
- id, client_id, eway_bill_no, invoice_no, trip_id
- vehicle_no, status, operation_type
- validity_end, route_status, api_provider, error_code
- from_place, to_place, from_poi_name, to_poi_name
- total_value, created_at, updated_at
```

### Table 2: client_operation_rules
```sql
Per-client configuration - determines behavior

Columns:
- client_id (UNIQUE)
- auto_generate, auto_part_b_update, auto_extend
- approval_required, gps_required, monitoring_only
- api_mode, invoice_source, retry_limit
- multi_vehicle_allowed, route_check_required
- active_flag
```

### Table 3: eway_operation_queue
```sql
Operation queue - what needs to be done

Columns:
- id, client_id, eway_bill_id, operation_type
- priority, status, retry_count, error_message
- queued_at, processed_at
```

### Table 4: eway_bill_logs
```sql
Audit trail - all actions logged

Columns:
- id, eway_bill_id, client_id, action
- old_status, new_status, metadata, created_at
```

### Table 5: eway_bill_exception_queue
```sql
Failed/manual cases - needs attention

Columns:
- id, client_id, eway_bill_id
- exception_type, message, severity
- escalated_to, created_at, resolved_at
```

### Scenario 2: Multi-Branch Same Client

**Problem:** Same client, different branches with different rules

**Solution:** Use composite key in config

```sql
-- Instead of: client_id (UNIQUE)
-- Use: (client_id, branch_code) as UNIQUE key

CREATE TABLE client_operation_rules (
  client_id TEXT,
  branch_code TEXT,
  operation_type TEXT,
  auto_generate INTEGER,
  ...
  UNIQUE(client_id, branch_code)
)

-- Example:
INSERT INTO client_operation_rules VALUES('ABC', 'branch1', 'AUTO', 1, ...);
INSERT INTO client_operation_rules VALUES('ABC', 'branch2', 'SEMI', 0, ...);
```

**Use in code:**
```javascript
const key = `${record.client_id}:${record.branch_code}`;
const rules = await getClientRulesByCompositeKey(key);
```

## Best Practices

### ✅ DO:

1. **Store behavior in database rules**
   ```javascript
   // ✓ Good
   const rules = await getClientRules(clientId);
   if (rules.auto_generate) generateEWB();
   ```

2. **Use universal statuses**
   ```javascript
   // ✓ Good
   // All clients use same status set
   const statuses = ['NEW', 'GENERATED', 'IN_TRANSIT', 'COMPLETED'];
   ```

3. **Log all actions**
   ```javascript
   // ✓ Good
   await logAction(ewbId, clientId, 'GENERATED', 'NEW', 'GENERATED', {...});
   ```

4. **Queue for approval**
   ```javascript
   // ✓ Good
   if (rules.approval_required) {
     await queueOperation(clientId, ewbId, 'WAITING_APPROVAL', 10);
   }
   ```

### ❌ DON'T:

1. **Hardcode per-client logic**
   ```javascript
   // ✗ Bad
   if (clientId === 'ABC') generate();
   if (clientId === 'XYZ') approve();
   ```

2. **Use different statuses per client**
   ```javascript
   // ✗ Bad
   // CLIENT_A uses 'APPROVED', CLIENT_B uses 'OK', CLIENT_C uses 'YES'
   ```

3. **Skip logging**
   ```javascript
   // ✗ Bad
   generateEWB(); // where's the audit trail?
   ```

4. **Ignore exceptions**
   ```javascript
   // ✗ Bad
   try { updatePartB(); } catch(e) { }
   ```
